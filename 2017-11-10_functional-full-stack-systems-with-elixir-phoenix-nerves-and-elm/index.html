Let's talk about why per-process garbage-collection matters. If we have a JVM that's under a constant workload, its memory usage profile will probably look like this. It grows for a while, then suddenly drops before growing again. The shaded areas are wasted memory because they were temporarily allocated to do some work, but weren't freed when they're no longer needed, because it's difficult to tell whether they're still in use somewhere else.
If we zoom in, we see that it is also continually doing this same clean-up process, but on a smaller scale, to keep memory usage in-check between Full-GCs. When the overall memory usage hits 75%, it triggers a Full-GC that stops the whole VM while it determines what memory is no longer referenced anywhere and frees it up. In a busy system with a lot of memory, this can take a considerable about of time and cause performance problems.
By contrast, the BEAM does garbage-collection on a per-process basis, so there's not much discernable memory-waste for a system under constant load. The overall memory usage only ebbs and flows with the actual memory requirements of your workload. This means that you can run a system closer to its memory limit, provided that your workload doesn't have large peaks and valleys. That can be a very important property for an embedded system with only half a gig of RAM.
If we zoom in, we see that this is due to each process allocating some memory as it does its work, garbage-collecting along the way, and freeing it all when it's finished. At any moment in time, there are a bunch of processes allocating a small increment of memory, while a much smaller number are freeing their entire allocation. So the net result is that the overall memory usage responds very quickly to the demand on the system.
